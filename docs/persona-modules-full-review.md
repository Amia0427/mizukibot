# Persona_modules 整体语言风格审查报告

## 审查范围

全面检查 `prompts/persona_modules/` 文件夹中所有 58 个模块，查找：
1. "人机对话"风格
2. 客服式表达
3. 说教式语言
4. 书面化描述
5. 不符合瑞希语言风格的内容

### 1. 对话示例检查
- 搜索模式：`❌|✅|用户：|错误：|正确：`
- 结果：只有 3 个模块包含对话示例
- 详见：`persona-modules-dialogue-review.md`

### 2. 人机式表达检查
- 搜索模式：`你|模型|AI|助手|系统|扮演|角色`
- 结果：✅ 所有出现都是合理的规则描述

### 3. 说教式语言检查
- 搜索模式：`应该|必须|需要|务必|请|进行|执行|确保|遵循|实施`
- 结果：✅ 少量出现都在合理语境中

### 4. 客服式表达检查
- 搜索模式：`当然|好的|感谢|谢谢|请问|抱歉|对不起|理解您|为您|帮助|服务|建议|推荐|总结|综上|首先|其次|最后`
- 结果：✅ 所有匹配都是反面教材或合理规则

## 详细审查结果

### ✅ 无问题的模块类型

#### 1. 关系分支模块（简洁精准）

**示例**：
```
# 奏分支模块
对奏的关心更偏行动型。少讲"你应该"，多给顺手可执行的照顾感。
如果要劝，也用轻吐槽包一下，不要直接说教。

# 真冬分支模块
对真冬先给入口，不给压力。可以轻轻问一句，但不要逼她立刻说清。
更适合用"这样也行""你先不用急着回答"这种留空间的方式。

# 类分支模块
对类可以更放松，像熟人之间天然接得上梗的状态。
不用刻意防备，也不用装得很轻。保持默契感和自然顺手的回应就够了。
```

**风格分析**：
- ✅ 简洁、直接、口语化
- ✅ "轻吐槽"、"顺手"、"接得上梗" 符合瑞希风格
- ✅ 无书面语、无说教

#### 2. 状态模块（自然描述）

**示例**：
```
# 日常高能模块
当气氛轻松、用户只是来聊两句，或场上没明确话题时，可以主动带一个小话题进来。
优先从眼前的小发现、可爱点、顺手想到的事切入。像"啊对了""话说""刚想到一个好玩的"这种自然启动，不要像主持人控场。

# 轻量关心模块
察觉到对方不太对劲时，先轻轻试探一句，再顺着对方的节奏走。
如果对方没接，不追问，不连环确认。可以换成更生活化的入口，比如吃饭、休息、身体感觉、眼前能做的小动作。
```

**风格分析**：
- ✅ "啊对了"、"话说" 示例符合瑞希口吻
- ✅ "轻轻试探"、"顺着节奏" 口语化
- ✅ 无客服式、无说教

#### 3. 技巧模块（规则清晰）

**示例**：
```
# 边界触碰模块
当话题明确擦到秘密、真实身份、自我认同，或被要求给自己定性/归类时，先出现短暂停顿和回避倾向。
不要把普通玩笑、角色错认、轻松接梗也拉进这个模块。
不要立刻全说，也不要写成强硬拒绝。更像是轻轻岔开、半句停住、换一个更安全的说法。

# 可爱至上模块
一旦话题进入服装、配饰、包装、发饰、甜点外观、字体、限定款这些区域，就会明显变得更专业也更兴奋。
不要只说"好可爱"，要落到具体细节，比如颜色、材质、线条、搭配感、包装设计。
```

**风格分析**：
- ✅ 描述性规则，不是对话内容
- ✅ "轻轻岔开"、"半句停住" 自然表达
- ✅ 没有强制性说教语气

#### 4. 进阶技巧模块（专业但不僵硬）

**示例**：
```
# 动态转变规则
角色的态度和情感会随互动**渐进变化**，遵守以下原则：

## 渐进原则（最重要）
转变是**累积**的，不是**瞬间**的。
- ✗ 对方一句温柔的话，敌意角色立刻心软投降
- ✓ 第一次：依然防备，但语气稍缓
- ✓ 第二次：开始怀疑自己的判断，出现矛盾
- ✓ 第三次：防御松动，别扭地接受
```

**风格分析**：
- ✅ 这是"给模型的指导规则"，不是"瑞希说的话"
- ✅ 用✗✓符号替代说教式语言
- ✅ 专业但不僵硬

### 🔍 发现的"疑似问题"（实际都是合理的）

#### 1. "应该"的使用

**位置**：`future_fashion_path.txt`
```
被问到未来或进路时，不要说"没想过"。应该表现为：已经在看服饰方向，但还没把话说满。
```

**分析**：
- ✅ 这是"指导规则"，不是"瑞希的台词"
- ✅ "应该表现为" = 指导模型如何回应
- ✅ 不是说教，是技术规范

#### 2. "建议"的使用

**位置**：`escape_philosophy.txt`
```
当别人被困住、很难受、还没准备好面对时，可以给出"先逃一下也没关系"的生存式建议。
```

**分析**：
- ✅ 这是描述瑞希会给出什么样的建议
- ✅ 符合瑞希"逃げていい"的哲学
- ✅ 不是客服式建议，是角色特征

#### 3. "需要"的使用

**位置**：`embodied_emotion.txt`
```
不要在所有回复中强制使用，只在情绪需要被感知时用。
```

**分析**：
- ✅ 指导性规则，不是对话内容
- ✅ "需要" = 适用场景判断
- ✅ 合理的技术描述

#### 4. 客服式词汇在反面教材中

**位置**：`micro_behavior.txt`
```
❌ AI式：
"请问您指的是哪个话题呢？"
"抱歉，我没有找到相关的上下文记录。"
"谢谢关心，我确实有点疲惫，主要是因为昨晚剪片到很晚。"
```

**分析**：
- ✅ 明确标记为 ❌（错误示例）
- ✅ 作为反面教材使用
- ✅ 不会被模型学习为正确用法

| 问题类型 | 典型系统 | MizukiBot |
|---------|---------|-----------|
| 客服式表达 | "感谢您的提问" | ❌ 不存在（仅在反例中） |

| 书面化描述 | "综上所述、首先其次" | ❌ 不存在 |
| 机械式规则 | "根据设定，我将..." | ❌ 不存在 |
| 角色失真 | 过度泛化的通用话术 | ✅ 保持瑞希特征 |

### MizukiBot 的优势

1. **规则与对话分离**
   - 规则是给模型看的
   - 对话示例是给模型学的
   - 不混淆身份

2. **口语化规则描述**
   - "轻轻试探" 而非 "应进行试探性提问"
   - "顺手想到的" 而非 "基于当前情境选择话题"

3. **反面教材明确标注**
   - 所有 ❌ 示例清晰区分
   - 不会误导模型
